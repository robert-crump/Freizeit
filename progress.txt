## 2026-07-16 — Issue #16: Distinct map pin styling (color/icon) for new POI categories

- Only open `ready-for-agent` issue this session — #15 (its blocker) closed
  already, unblocking it.
- Read `CategoryUi.kt`, `PoiMap.kt`, `MapStyle.kt`/`categoryColorExpression()`,
  and `ExploreScreen.kt`'s `CategoryDot`/`LayersPanel`/`PoiRow` before writing
  anything. Found the whole per-category-color path is already data-driven off
  one list: `CATEGORY_ORDER` (used for both sort order and as the source of
  `categoryColorExpression()`'s match stops) only listed the original 5, so
  `categoryColor()` silently fell back to the shared gray for all 6 new
  categories everywhere at once — the map circle layer, `CategoryDot` in the
  list/panel, and the Layers panel's category rows. This matches #15's own
  note that this was "the expected pre-#16 state, not a bug." So the fix is
  two `CategoryUi.kt` edits, not three separate ones: extending
  `CATEGORY_ORDER` alone fixes `categoryColorExpression()`, `CategoryDot`, and
  `LayerRow` simultaneously since they all read through `categoryColor()`/
  `categoryOrderIndex()`. No `PoiMap.kt` change needed — its circle-color
  property already reads `categoryColorExpression()` verbatim.
- `CATEGORY_ORDER` extended to all 11, ordered to match the extraction
  registry's `CATEGORIES` list in `extract_pois.py` (`EXACT_CATEGORIES` then
  `COARSE_CATEGORIES`: shop, tourism, leisure_other, office, craft, historic),
  matching the doc comment's existing claim that this list "match[es] the
  extraction registry."
- `categoryColor()`: 6 new hex colors, chosen by computing HSL hues spread
  across the gaps left by the original 5 (orange 32°, green 123°, brown ~17°,
  red 0°, pink 336°) so every pair of the 11 stays ≥28° apart on the color
  wheel even with all shown together in "All POIs" mode: shop `#1B7D98`
  (blue-cyan, 193°), tourism `#1B986A` (teal, 158°), leisure_other `#7CA51D`
  (olive/yellow-green, 78°, sitting in the wide gap between orange and green),
  office `#1D38A5` (blue-indigo, 228°), craft `#511DA5` (violet, 263°),
  historic `#A01DA5` (purple-magenta, 298°, 38° clear of pink).
- Interpreted "icon" in the issue title/body as the existing dot-style — this
  codebase has no per-category image/sprite icon system for any of the
  original 5 categories either (`CategoryDot` is a plain colored circle,
  `PoiMap.kt`'s pins are a `CircleLayer`, not a `SymbolLayer` with images), so
  adding one now would be new infrastructure beyond "following the existing
  pattern in `CategoryUi.kt`" that the issue explicitly asked to follow. Flagging
  as a possible future issue if real icon glyphs (not just color) turn out to
  matter once someone eyeballs the map with all 11 categories on screen.
- Cluster circles/count labels needed no change: `clusterCircleLayer()` colors
  every cluster from `MaterialTheme.colorScheme.secondary` regardless of the
  categories inside it (count-based size tiers only), so mixed-category
  clusters were already category-agnostic and stay legible by construction —
  confirmed by reading `PoiMap.kt` rather than assuming.
- Added `CategoryUiTest.kt` (new file, `ui/common` had no test coverage
  before): asserts `CATEGORY_ORDER` is exactly the 11 in extraction-registry
  order, all 11 colors are pairwise distinct (`.toSet().size` check — would
  catch an accidental hex collision even if hue math above has a bug), the 6
  new categories don't hit the shared fallback gray, and they sort after
  `ice_cream` and before an unknown category.
- Tests: 104 total (was 100), 23 failed (was 23) — all 4 new `CategoryUiTest`
  cases green; the 23 failures are the pre-existing `PoiImportTest`
  Robolectric `UnsatisfiedLinkError` set from #15/#13/#10 (`MapLibre.getInstance()`

## 2026-07-15 — Issue #14: Widen POI extraction to shop/tourism/leisure_other/office/craft/historic

- Picked over #15/#16 since both are blocked transitively on this one (#15
  directly, #16 via #15) — this was the only unblocked ready-for-agent issue,
  and this machine has the Ride-Graph `.pbf` sources + `osmium` venv the
  issue's note flags as a hard requirement.
- `extract_pois.py`: split the old single `CATEGORIES` dict into
  `EXACT_CATEGORIES` (the original 5 — playground/park/cafe/restaurant/
  ice_cream, unchanged key=value matching, no name requirement) and a new
  `COARSE_CATEGORIES` (shop/tourism/leisure_other/office/craft/historic —
  matches any object carrying the given top-level OSM key regardless of
  value, but only when it also has a non-empty `name` tag). `categorize()`
  checks exact first, then coarse, so e.g. `leisure=playground` still
  resolves to "playground" and never falls into `leisure_other`
  (`_LEISURE_EXACT_VALUES` exclusion). The name requirement was in the
  issue's acceptance criteria, to keep the coarse keys from pulling in
  unnamed street furniture (benches, vacant shops, etc.) — confirmed
  necessary by the coverage report below.
  `filter_keys` for `osmium`'s `KeyFilter` widened to the union of both
  dicts' keys so the new categories actually get read from the .pbf stream.
- `README.md` updated: documents the exact-vs-coarse split, the name-tag
  rule and why it exists, and the expanded `categories` field in the output
  example.
- Ran via the Ride-Graph venv against the real full extracts (koeln-regbez,
  limburg, wallonia_french/german — not `precut/`). `data/pois.json`
  regenerated in place (git-ignored, not committed). Coverage report:
  playground 9635 (14.1% named), park 4466 (28.5%), cafe 3037 (94.7%),
  restaurant 10145 (97.6%), ice_cream 612 (96.4%) — all five unchanged in
  shape from before this issue, confirming the exact-match path wasn't
  touched. New categories, all 100% named by construction: shop 45772,
  tourism 16904, leisure_other 9004, office 11450, craft 3745, historic
  15955. 130725 POIs total (2749 cross-extract duplicates dropped), up from
  ~28k pre-#14.
- No test harness exists for this Python tool (none did before either) and
  no app/Kotlin code was touched, so no Gradle run — this issue is data/
  tooling-only. #15 (Room import/query) and #16 (map pin styling) are the
  ready-for-agent issues that consume this new data next.
